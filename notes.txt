=== Kimis System Technical Summary ===

1. INTRODUCTION
This program is an Express.js backend server that provides a RESTful API for handling user authentication and administrative data. It securely interacts with a PostgreSQL database, tracks users through stateless JSON Web Tokens (JWTs), and resolves CORS (Cross-Origin Resource Sharing) barriers to allow a distinct frontend app (HTML/JS) to make requests seamlessly.

2. DATABASE IMPLEMENTATION (PostgreSQL)
The application originally utilized a dummy memory-based array to mock user records. It has now been upgraded to a production-ready RDBMS:
- Database Driver: Uses the `pg` (node-postgres) package.
- Connection: Connected to PostgreSQL natively on `localhost:5432` against user `postgres` with password `11111`.
- The Database: Targets a database named `kimis`.
- The Table: It expects a table named `users` with standard fields: `id` (SERIAL PRIMARY KEY), `email` (VARCHAR UNIQUE), `password` (VARCHAR), and `role` (VARCHAR DEFAULT 'user').

To keep things efficient, we don't construct and connect single queries one at a time. The database configuration utilizes a robust `pg.Pool` connection pool. 
- A Pool acts safely like a queue for client queries, efficiently handling concurrent requests matching users up with idling database connections, freeing up resources when queries settle.

Prepared Statements / Parameterization: 
- To avoid highly dangerous "SQL Injections," we never interpolate untrustworthy user variables directly into query strings (like "SELECT * FROM users WHERE email = " + email).
- Instead, the program sends parameterized queries like ($1, $2). The `pg` module safely binds these input values underneath the hood. 
Example query: `pool.query("SELECT * FROM users WHERE email = $1", [email])`

3. AUTHENTICATION (JSON Web Tokens - JWT)
The system leverages JWTs (JSON Web Tokens) with a symmetric signature algorithm ("your_super_secret_key" mapping to `SECRET` variable).
- How Login Works (`POST /api/login`): 
  You provide the program with an email and a password. A PostgreSQL query validates the record against the user table. If successful, `jwt.sign()` generates a 1-hour expiring security token wrapped directly in a JSON packet that gets handed back to the user's browser.
  
- How JWTs are Checked: We use an architectural pattern called "Middleware" (`function authenticateToken`).
  Whenever users try to hit guarded endpoints like retrieving their personal info (`GET /api/me`) or hitting admin routes (`GET /api/admin-data`), their web requests must have a header specifying: `Authorization: Bearer <the-provided-token>`.
  The middleware halts their request, cryptographically validates the token payload utilizing the exact same `SECRET` key used to originally mint it, blocks entry if tampered/hacked, and moves to the next application instruction on verify passing.

4. ENDPOINT OVERVIEW
The main exposed routing pathways over Port 8080 look like this:
- `POST /api/login`: Looks up ($1, $2) email alongside password into database, generating a token.
- `POST /api/register`: Checks against duplicates ($1), natively inserts via PostgreSQL syntax into the `users` table via `INSERT INTO users (email, password, role) VALUES ($1, $2, 'user')`.
- `GET /api/me`: Enforces Middleware Authorization (`authenticateToken`), looking up your payload Token ID into DB.
- `GET /api/admin-data`: Enforces dual Middleware logic (`authenticateToken` > `authorizeRole('admin')`), triggering a `SELECT COUNT(*)` on active users from PG exclusively if valid.
